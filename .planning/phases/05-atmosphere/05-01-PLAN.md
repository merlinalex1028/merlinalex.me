  <name>Task 1: Create atmosphere gate utility + AtmosphereLayer wrapper</name>
  <files>src/lib/atmosphere.ts, src/components/atmosphere/AtmosphereLayer.astro, src/layouts/BaseLayout.astro</files>
  <action>
Create `src/lib/atmosphere.ts` — the shared atmosphere gate utility consumed by all Phase 5 islands. This module wraps the `window.__atmo__` global API (set by the BaseLayout pre-paint script) into typed helper functions.

The module must export these functions:
- `isOff(): boolean` — returns `true` when `window.__atmo__?.level === 'off'` or when `__atmo__` is unavailable
- `isSubtle(): boolean` — returns `true` when level is `'subtle'`
- `isFull(): boolean` — returns `true` when level is `'full'`
- `isReducedMotion(): boolean` — returns `true` when `window.__atmo__?.reducedMotion === true` OR when `matchMedia('(prefers-reduced-motion: reduce)').matches` (defensive: the pre-paint script sets `reducedMotion` but islands should also check the live media query)
- `isMobile(): boolean` — returns `true` when `window.innerWidth <= 768`
- `onAtmoChange(fn: (state: { level: string; theme: string; reducedMotion: boolean }) => void): () => void` — subscribes to `window.__atmo__` changes via `subscribe()`, returns an unsubscribe function that removes the listener from `_listeners`
- `getLevel(): 'off' | 'subtle' | 'full'` — returns current effective level

All functions must guard against `window.__atmo__` being `undefined` (SSR safety — these only run client-side but defensive coding prevents runtime crashes if the pre-paint script failed).

The `onAtmoChange` unsubscribe function must splice the listener from `window.__atmo__._listeners` array (not just no-op), so islands that destroy themselves don't leak listeners.

Create `src/components/atmosphere/AtmosphereLayer.astro` — a fixed-position wrapper component that hosts all atmosphere islands. This component:
- Renders a `<div class="atmo-layer">` with `position: fixed; inset: 0; pointer-events: none; z-index: 50;`
- `pointer-events: none` on the container, but child islands set `pointer-events: auto` on their interactive elements (so the layer doesn't block page scroll/clicks)
- The container is hidden via the existing CSS gate: `:root[data-atmo="off"] .atmo-layer { display: none; }` — add this rule to `src/styles/global.css` alongside the existing `.atmo-petals` / `.atmo-cursor-trail` / `.atmo-live2d` gates
- Contains a `<slot />` for child islands
- Has no `client:` directive itself — it's a plain Astro component (static HTML wrapper). Child islands use `client:idle` individually.

Modify `src/layouts/BaseLayout.astro`:
- Import `AtmosphereLayer` from `'../components/atmosphere/AtmosphereLayer.astro'`
- Render `<AtmosphereLayer><slot name="atmosphere" /></AtmosphereLayer>` just before the closing `</body>` tag
- Add a named slot `atmosphere` so pages can inject atmosphere islands (or, for the MVP, just render the layer with default children — the islands in plans 05-02 through 05-05 will be added directly inside the layer)

For the MVP approach: render the AtmosphereLayer with a default slot. Plans 05-02 through 05-05 will add their island components inside this layer by modifying BaseLayout.astro to import and render them.

<task type="auto">
  <name>Task 2: Add date-based atmosphere CSS tweaks</name>
  <files>src/styles/global.css</files>
  <action>
Add lightweight date-based CSS atmosphere tweaks to `src/styles/global.css`. These are pure CSS — no JavaScript, no build-time logic. They apply via CSS custom properties that override accent colors on specific calendar dates.

Add a new section at the end of `global.css` (after the reduced-motion baseline) with the comment `/* DATE-BASED ATMOSPHERE TWEAKS — Phase 5 */`.

Create 3 date-based tweaks using CSS `@media` with `prefers-color-scheme` for theme awareness:

1. **Valentine's Day / White Day (Feb 14 + Mar 14)**: Override `--color-accent` to a deeper rose (`#E84393`) and `--color-accent-subtle` to a warm pink (`#FD79A8`). Use a CSS class `.atmo-date-valentine` that can be applied to `<html>` via a tiny inline script checking the date.

2. **Christmas season (Dec 20-25)**: Override `--color-accent` to a festive red-green gradient feel — accent becomes `#C0392B` (warm red), accent-subtle becomes `#27AE60` (pine green). Use class `.atmo-date-christmas`.

3. **Sakura season (Mar 25 - Apr 10)**: Override `--color-accent` to a lighter cherry blossom pink (`#FFB7C5`), accent-subtle to `#FFE4ED`. Use class `.atmo-date-sakura`.

Implementation approach:
- Add CSS rules scoped to `.atmo-date-valentine`, `.atmo-date-christmas`, `.atmo-date-sakura` on `:root` that override the accent CSS custom properties
- These rules ONLY override `--color-accent` and `--color-accent-subtle` — they don't touch layout, spacing, or other tokens
- The classes are applied by a small inline `<script>` in BaseLayout.astro's `<head>` (after the pre-paint script) that checks `new Date()` and adds the class to `document.documentElement` if the date matches. This script must be `<script is:inline>` (synchronous, blocking) so the class is set before first paint — same pattern as the theme/atmo pre-paint script.

Add to BaseLayout.astro after the existing pre-paint script block:
```
<script is:inline>
(function(){
  try {
    var d = new Date();
    var m = d.getMonth() + 1; var day = d.getDate();
    if ((m === 2 && day === 14) || (m === 3 && day === 14)) {
      document.documentElement.classList.add('atmo-date-valentine');
    } else if (m === 12 && day >= 20 && day <= 25) {
      document.documentElement.classList.add('atmo-date-christmas');
    } else if ((m === 3 && day >= 25) || (m === 4 && day <= 10)) {
      document.documentElement.classList.add('atmo-date-sakura');
    }
  } catch(e) {}
})();
</script>
```

This script is intentionally tiny (<10 lines) and runs synchronously in `<head>` so the date class is available before any styles compute.
