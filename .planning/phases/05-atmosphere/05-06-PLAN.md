<task type="auto">
  <name>Task 1: Fix PetalsIsland — tsParticles container target</name>
  <files>src/components/atmosphere/PetalsIsland.astro</files>
  <action>
Fix `src/components/atmosphere/PetalsIsland.astro`. The root cause is that `tsParticles.load()` is called with `{ id: 'tsparticles-petals' }` where the target element is a `<canvas>`. tsParticles expects a container `<div>` — it creates its own canvas inside the container. When given a canvas element, it creates a nested canvas with zero dimensions.

Changes required:

1. Replace the `<canvas id="tsparticles-petals">` element with a plain `<div>`:
   ```
   Before: <canvas id="tsparticles-petals"></canvas>
   After:  <div id="tsparticles-petals" style="width:100%;height:100%;"></div>
   ```
   Keep the outer `.atmo-petals` wrapper with `pointer-events: none`.

2. In the `initParticles()` function, ensure `tsParticles.load()` targets the new div container. The call should remain `tsParticles.load({ id: 'tsparticles-petals', options: { ... } })` — this now targets the div instead of the canvas.

3. Add a defensive check: after `tsParticles.load()`, verify the container has a canvas child. If not, log a warning:
   ```
   const container = document.getElementById(CONTAINER_ID);
   if (container && !container.querySelector('canvas')) {
     console.warn('PetalsIsland: tsParticles did not create a canvas');
   }
   ```

4. Verify the sakura.svg asset loads correctly: the `image.src` is `/petals/sakura.svg`. Confirm `public/petals/sakura.svg` exists (it does — verified during planning).

Do NOT change the particle configuration (colors, speed, count, wobble, etc.) — only fix the container target.
  </action>
  <verify>
    <automated>pnpm build 2>&1 | tail -5</automated>
  </verify>
  <done>
    - PetalsIsland.astro uses a `<div>` container (not `<canvas>`) for tsParticles.load()
    - `pnpm build` exits 0
    - tsParticles creates its own canvas inside the container div
  </done>

   - In `spawnPetal()`, add a debug log for the first spawn only: `console.debug('CursorTrail: spawning petal at', x, y)`

   ```
   const accentColor = getComputedStyle(document.documentElement).getPropertyValue('--color-accent').trim() || '#FF6B9D';
   ```

3. **Verify z-index stacking**: The cursor trail particles use `z-index: 9999` via the `.cursor-petal` class. The `AtmosphereLayer` uses `z-index: 50`. Since the particles are appended to `document.body` (not inside AtmosphereLayer), they should render above it. Verify this is correct — if the particles are somehow being clipped by a parent, add `position: relative` to body or ensure no `overflow: hidden` on a parent element clips them.

4. **Check if the script runs at all**: The most likely issue is that `isOff()` returns `true` because `window.__atmo__` is not set when the script runs.   
   ```
   function waitForAtmo(callback: () => void): void {
     if ((window as any).__atmo__) {
       callback();
     } else {
       // Fallback: wait for atmo to be set
       const check = setInterval(() => {
         if ((window as any).__atmo__) {
           clearInterval(check);
           callback();
         }
       }, 50);
       // Timeout after 2 seconds
       setTimeout(() => clearInterval(check), 2000);
     }
   }
   ```
   Wrap the initial gate logic in `waitForAtmo()` to handle any race condition.

5. **Remove the debug logging** (from steps 1) after verifying the trail works — or keep it behind a `location.search.includes('debug=cursor')` guard for future debugging.
